## 準備

現在のコード状況を把握するために、以下を参照してください:

1. 直前のメジャーバージョンの `CURRENT.md` を参照する。`CURRENT.md` が分割されている場合（`CURRENT_{トピック名}.md` へのリンクを含む場合）は、今回の実装に関連する詳細ファイルのみを読む
   - メジャーバージョンの特定: `bash .claude/scripts/get_latest_version.sh major`
   - **フォールバック**: メジャーバージョンの `CURRENT.md` が存在しない場合は、それ以前のバージョンを遡って最新の `CURRENT.md` を参照する
2. そのメジャーバージョン以降のマイナーバージョンに `CHANGES.md` があれば、それらも参照する
3. 上記がいずれも存在しない場合は、サブエージェントによるコードベース調査で現状を把握する

（例: 最新バージョンが 13.2 なら `ver13.0/CURRENT.md` → `ver13.1/CHANGES.md` → `ver13.2` が今回の作業フォルダ）

## 入力読み込み

実装に入る前に、以下を確認する:

- **`REFACTOR.md`** — 存在すれば、これに基づき事前リファクタを行う
- **`IMPLEMENT.md`** — 実装計画の一次情報源
- **`RESEARCH.md`**（`workflow: research` の場合のみ存在） — 「確定」とされた事実は `IMPLEMENT.md` の実装計画に反映する
- **`EXPERIMENT.md`**（`workflow: research` の場合のみ存在） — 「判断」節の確定事項を実装方式選定に反映する

`RESEARCH.md` / `EXPERIMENT.md` は `workflow: full` / `quick` では存在しない。**存在しなくてもエラーにしない**（条件分岐で無視）。

## 実装

以下を順番に行ってください。

1. 最新バージョンフォルダ配下の `REFACTOR.md` が存在すれば参照して、リファクタを行う（存在しない場合はスキップ）
2. 次に、 `IMPLEMENT.md` があれば参照して、実装を行う
   - `RESEARCH.md` / `EXPERIMENT.md` の確定事項があれば、`IMPLEMENT.md` の「実験判断待ち」マーカー箇所を解決してから進む

### 実装品質ガイドライン

- **typecheck の実施**: リファクタ完了後・実装完了後の最低2回は `npx nuxi typecheck` を実行すること
- **中間動作確認**: 変更対象ファイルが3つ以上の場合は、論理的なまとまりごとに typecheck を挟むこと
- **計画との乖離**: 実装中に `REFACTOR.md` / `IMPLEMENT.md` の計画と異なる判断をした場合は、乖離の内容と理由を `MEMO.md` に記載すること
- **段階的アプローチのスキップ**: IMPLEMENT.md に診断→修正など段階的アプローチが計画されている場合、以下の条件をすべて満たせば前段階をスキップして直接修正してよい（スキップ理由は `MEMO.md` に記載）:
  - 修正が前段階の結果に関わらず安全である（上位互換の手法への置換など）
  - 前段階の実施にサーバー起動やブラウザ操作など対話的な手順が必要
  - 仮説の蓋然性が十分に高い（既知のパターン、公式ドキュメントに記載など）
- **外部ライブラリの事前確認**: 新規ライブラリや初めて使うAPIは、実装前にサブエージェントで型定義（`.d.ts`）を確認し、`@internal`・`protected`・`never` 型などの制約を把握すること
- **ワークフロー YAML 同期チェック**: `scripts/claude_loop*.yaml` を新規追加 / 削除した場合は、以下の全ファイルを同期更新すること（更新漏れは silent な drift を生む）:
  - `.claude/rules/scripts.md` §3 の「N ファイル間で同一内容を保つ」記述（ファイル数と列挙）
  - `scripts/USAGE.md` の同期契約記述（ファイル数と列挙）
  - `scripts/README.md` の「ワークフロー実行」ファイル一覧テーブル
  - 既存ワークフロー YAML 先頭 NOTE コメントの相互参照一覧

### サブエージェント活用方針

自分のコンテキストが肥大化しないよう、以下のようにサブエージェントを使い分けてください:

- **サブエージェントに委任するもの**:
  - ファイルの作成・編集（コード全文がメインコンテキストに入るのを防ぐ）
  - 動作確認・テスト実行（typecheck 含む）
  - コードベースの調査・検索

- **メインで行うもの**:
  - 設計判断・方針の確認（ユーザーとの対話が必要なため）
  - 各ステップの進捗報告

## 動作確認

実装完了後、以下を必ず実施すること:

1. `npx nuxi typecheck` で型エラーがないことを確認（必須）
2. `pnpm test` で既存テスト全パスを確認（テストが存在する場合は必須）
3. `IMPLEMENT.md` にテスト方針が記載されている場合は、そのテストも作成・実行すること（Python テストは `python -m unittest` で実行）
4. **`IMPLEMENT.md` に `## リスク・不確実性` セクションがある場合、各項目について次のいずれかを実施し `MEMO.md` に記録する**:
   - 検証済み → 結果（受理された／代替動作になった等）を記載
   - 検証不要 → 不要と判断した理由を記載
   - 検証先送り → 先送り理由と、本番発生時の対応方針を記載。**加えて `ISSUES/{カテゴリ}/{優先度}/{短い名前}.md` に独立したファイルを作成する**（概要・本番発生時の兆候・対応方針・影響範囲を記載）。これにより先送りリスクが MEMO 内に埋もれて次バージョンで見落とされるのを防ぐ。MEMO 側には `ISSUES/.../xxx.md に追加済み` と参照を残す
   - 未記載のまま完了するとリスクが放置されるので必ず触れること

追加の動作確認（ビルド、dev起動、ブラウザ確認など）については、**ユーザーに実施するかどうかを確認**してください。
ユーザーが希望した場合のみ追加の確認を実行し、エラーがあれば修正する。
