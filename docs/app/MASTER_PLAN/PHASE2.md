# PHASE2: 機能拡張（PHASE1からの差分）

## 概要

PHASE1の基盤に対し、以下3つの軸で拡張する:
1. **永続化とマルチスレッド** — SQLiteで会話履歴をサーバー側に保存し、複数スレッドを管理可能にする
2. **Web検索連携** — Tavily Searchで最新の類似事例をリアルタイム取得
3. **フロー制御の厳密化** — LangGraphステートマシンで対話ステップを明示的に管理

## 対話フローの変更点

```
PHASE1: 全ステップをLLMのプロンプト制御に委ねていた
         ↓
PHASE2: LangGraphステートマシンでステップを明示的に分離

[課題入力] → [抽象化ノード] → [事例検索ノード] → [事例提示] → [ユーザー選択] → [解決策生成ノード]
                                      │
                                      ├── LLM内部知識
                                      └── Tavily Search（Web検索）
```

- 各ノードが独立したシステムプロンプトを持ち、役割を厳密に限定
- ステップ3でLLM内部知識とWeb検索結果の両方を統合して事例を提示

## やること

### 1. 会話履歴の永続化（SQLite）

- サーバー側にSQLiteを導入し、会話履歴を永続化
- PHASE1のインメモリ MemorySaver を置き換え
- サーバー再起動後も会話を復元可能にする

### 2. 複数スレッド管理

- サイドバーにスレッド一覧を表示
- 「新規チャット」ボタンでスレッドを作成
- AIが会話内容からスレッドタイトルを自動生成
- スレッド切り替えで会話履歴をロード

### 3. Web検索連携（Tavily Search）

- LangChainのTavily Search Integrationを利用
- ステップ3（類似事例提示）でWeb検索結果をLLM内部知識と統合
- AIが自動判断で検索を実行（ユーザーの明示的な指示は不要）

### 4. LangGraphステートマシンによるフロー制御

- PHASE1: 単一のシステムプロンプトでLLMに全ステップを委任
- PHASE2: ステートマシンで各ステップをノードとして分離
  - **抽象化ノード**: 課題を抽象化する専用プロンプト
  - **事例検索ノード**: 抽象化結果を元にLLM知識 + Web検索で事例を収集
  - **事例提示ノード**: 収集した事例をユーザーに提示
  - **解決策生成ノード**: ユーザー選択を受けて最終回答を生成
- ノード間の状態遷移を明示的に定義し、フローの再現性を確保

## やらないこと（PHASE1から継続）

- 認証・ユーザー管理
- RAG構成・事例データベース
- モバイル最適化
- デプロイ・本番運用

## 構成の変更点

```
PHASE1:
  API Route → LangChain → OpenAI API

PHASE2:
  API Route → LangGraph (ステートマシン)
                ├── 抽象化ノード → OpenAI API
                ├── 事例検索ノード → OpenAI API + Tavily Search API
                └── 解決策生成ノード → OpenAI API
              └── SQLite (会話履歴永続化)
```
