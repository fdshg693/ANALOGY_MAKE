# PHASE3: ユーザーカスタマイズ & 会話分岐（PHASE2からの差分）

## 概要

PHASE2の完成形に対し、以下4つの軸で拡張する:
1. **動的設定システム** — AI回答の粒度・検索パラメータをスレッド内でリアルタイムに変更可能にする
2. **検索結果の可視化** — Tavily検索の生データをUI上で確認可能にする
3. **会話分岐（メッセージ編集）** — 過去メッセージを書き換えて会話を分岐、分岐間を自由に行き来
4. **動作確認便利化** — エコーモード・システムプロンプト変更で開発時の動作確認を効率化

## 1. 動的設定システム

### 1.1 AI回答の粒度設定

スレッド内の任意のタイミングで、AIの回答スタイルを切り替え可能にする。

**プリセット（ワンクリック選択）:**

| プリセット | 説明 | システムプロンプトへの影響 |
|---|---|---|
| 簡潔 | 箇条書き中心、要点のみ | 「簡潔に箇条書きで回答してください」を追加 |
| 標準 | 現在のデフォルト動作 | 追加指示なし |
| 詳細 | 具体例・背景説明を含む丁寧な回答 | 「具体例と背景説明を含めて詳しく回答してください」を追加 |

**カスタム指示:**
- プリセットに加え、自由テキストで追加指示を入力可能（例:「英語で回答して」「コード例を多めに」）
- カスタム指示はプリセットの指示に**追加**される

**適用スコープ:**
- 設定はスレッドごとに保持（同一スレッド内で変更しても他スレッドに影響しない）
- 設定変更は即座に次のAI応答から反映（過去の応答は変わらない）

**実装方針:**
- `threads` テーブルに設定カラムを追加（JSON形式: `settings TEXT`）
- 各ノードのシステムプロンプトに設定を動的注入
- フロントエンドに設定パネルUI（チャットエリア上部 or サイドパネル）

### 1.2 検索設定

Tavily Web検索のパラメータをユーザーが制御可能にする。

| パラメータ | 選択肢 | デフォルト |
|---|---|---|
| Web検索 ON/OFF | トグル | ON（APIキーがある場合） |
| 検索深度 | basic / advanced | basic |
| 取得件数 | 1〜10（スライダー or セレクト） | 3 |

**適用スコープ:** AI回答粒度と同様、スレッドごとに保持

**実装方針:**
- スレッド設定JSON内に `search` フィールドを追加
- `performSearch()` が設定値を参照してTavilyパラメータを動的に変更
- 検索 OFF 時は `caseSearchNode` がLLM内部知識のみで事例生成

## 2. 検索結果の可視化

Tavily検索の生データをチャットUI上で確認可能にする。

**表示方式:**
- AIの事例提示メッセージに、折りたたみセクション（`<details>`相当）で検索結果を付与
- 折りたたみ時: 「🔍 Web検索結果（N件）」のラベルのみ
- 展開時: 各結果のタイトル・URL・スニペットを表示

**データフロー:**
```
caseSearchNode
  ├── Tavily検索実行 → 生データを取得
  ├── 生データをstateに保存（新フィールド: searchResults）
  ├── 生データ + LLM知識 → LLMが事例を合成
  └── SSEで事例テキスト + 検索メタデータの両方を送信

フロントエンド
  └── 事例メッセージに検索メタデータを付与して表示
```

**実装方針:**
- `AnalogyState` に `searchResults` フィールドを追加
- SSEイベントに `search-results` タイプを追加（検索メタデータ送信用）
- `ChatMessage.vue` で検索結果の折りたたみ表示コンポーネントを追加
- 検索なし時（OFF or APIキー未設定）はセクション自体を非表示

## 3. 会話分岐（メッセージ編集）

過去のメッセージを編集して会話を分岐させる。元の分岐に戻ることも可能。

### 3.1 ユーザー操作フロー

```
1. 過去のユーザーメッセージにホバー → 「編集」ボタン表示
2. メッセージを編集して送信
3. 編集箇所以降の会話が新しい分岐として生成される
4. メッセージ横に分岐ナビゲーション（◀ 1/3 ▶）が表示される
5. ナビゲーションで分岐を切り替え可能
```

### 3.2 UI イメージ

```
┌───────────────────────────────────────┐
│  [User] 新幹線の騒音問題      [編集] │
│                                       │
│  [AI]   抽象化すると...               │
│                                       │
│  [User] カワセミの事例で  ◀ 1/2 ▶    │
│         ↑ 分岐あり: 2つの会話履歴     │
│                                       │
│  [AI]   解決策: ...                   │
└───────────────────────────────────────┘
```

### 3.3 データモデル

LangGraphのチェックポイントシステムを活用した分岐管理:

- LangGraph の `SqliteSaver` は各ステップのチェックポイントを保存済み
- 分岐 = 特定チェックポイントから新しい `thread_ts`（タイムスタンプ）で再実行
- 分岐メタデータは別テーブル `thread_branches` で管理

```sql
CREATE TABLE thread_branches (
  branch_id TEXT PRIMARY KEY,
  thread_id TEXT NOT NULL,
  parent_branch_id TEXT,          -- 親分岐（NULLならメイン）
  fork_message_index INTEGER NOT NULL, -- 分岐元のメッセージインデックス
  checkpoint_id TEXT NOT NULL,    -- LangGraphのチェックポイントID
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (thread_id) REFERENCES threads(thread_id)
);
```

**実装方針:**
- メイン会話 = `branch_id: "main"` として暗黙管理
- メッセージ編集時: 分岐元のチェックポイントを取得 → 新ブランチを作成 → 編集済みメッセージで再実行
- 分岐切り替え: フロントエンドで `branch_id` を切り替え、対応するチェックポイントから履歴をロード
- API: `POST /api/chat/fork`（分岐作成）、`GET /api/chat/branches`（分岐一覧取得）

## 4. 動作確認便利化

ISSUE `app/medium/動作確認便利化.md` への対応。

### 4.1 エコーモード

AIを使わず、ユーザー入力をそのまま返すモード。UI動作の確認に使用。

- 設定パネルのモード切り替えで有効化（AI回答粒度の上位設定）
- エコーモード時は LangGraph を経由せず、サーバー側で即座にユーザー入力をSSE返却
- ストリーミング表示の動作確認にも利用可能（文字を少しずつ返す）

### 4.2 システムプロンプトのカスタマイズ

- 設定パネルに「システムプロンプト上書き」テキストエリアを追加
- 設定時: 各ノードのデフォルトプロンプトの**前に**カスタムプロンプトを追加
- 空欄時: デフォルトプロンプトのみ（現在の動作）
- 本番環境では非表示（`process.dev` で制御）

## 構成の変更点

```
PHASE2:
  API Route → LangGraph (固定プロンプト) → OpenAI API / Tavily Search
              └── SQLite (会話履歴)

PHASE3:
  API Route → LangGraph (動的プロンプト + 設定注入)
                ├── 抽象化ノード → OpenAI API（粒度設定反映）
                ├── 事例検索ノード → OpenAI API + Tavily Search（検索設定反映）
                │                    └── 検索メタデータをstateに保存
                ├── 解決策生成ノード → OpenAI API（粒度設定反映）
                └── エコーモード → OpenAI APIスキップ、入力をそのまま返却
              └── SQLite
                    ├── 会話履歴（チェックポイント）
                    ├── スレッド設定（JSON）
                    └── 分岐メタデータ（thread_branches）
```
