# ver12: Web検索連携（Tavily Search）

## 対応方針

MASTER_PLAN PHASE2 の項目3「Web検索連携（Tavily Search）」に対応する。
ISSUES/app/high に未解決の課題がないため、MASTER_PLAN の次の機能追加を進める。

## 概要

現在のエージェントはLLMの内部知識のみで類似事例を提示している。Tavily Search API を LangChain ツールとして組み込み、ステップ3（類似事例の提示）でWeb検索結果を内部知識と統合して提示できるようにする。

## 提供される機能

1. **Web検索ツールの統合**: 既存の `createAgent` に Tavily Search ツールを追加し、AIがステップ3でWeb検索を活用して類似事例を取得できるようにする
2. **プロンプトの拡張**: システムプロンプトを更新し、ステップ3でWeb検索ツールの活用を強く推奨する指示を追加する
3. **環境変数の追加**: `NUXT_TAVILY_API_KEY` を Nuxt runtimeConfig に追加し、Tavily API キーをサーバーサイドで安全に管理する

## 検索実行の動作定義

Web検索の実行は **ReActエージェントの自律判断** に委ねる。システムプロンプトでステップ3での検索ツール活用を強く推奨するが、最終的にはAIが文脈に応じて検索の要否を判断する。「ステップ3で必ず検索を実行する」という強制は、現在のReActエージェント構成では保証できないため要件としない。

ステップ3での検索実行を強制する仕組みが必要になった場合は、ver13以降のLangGraphステートマシン移行で「事例検索ノード」として実装する。

## ver13 LangGraph移行との関係

MASTER_PLAN PHASE2 では、Tavily Search は LangGraph ステートマシンの「事例検索ノード」に組み込まれる設計になっている。ver12 ではあえてステートマシン移行と分離し、既存の ReactAgent にツールを追加する形で実装する。

**分離の理由:**
- Tavily API の統合・動作確認を段階的に行い、ステートマシン移行と同時に行うリスクを回避する
- APIキーの疎通確認、検索結果の品質確認、プロンプトチューニングを先行して実施できる
- ver12 の Tavily ツール統合コードは ver13 で置き換えられることを意図的に受容する（変更箇所は `analogy-agent.ts` のツール配列とプロンプト程度で、廃棄コストは小さい）

## スコープ

- ReActエージェントの自律判断でWeb検索を実行する（ユーザーの明示的な指示は不要）
- 検索結果はLLM内部知識と統合して類似事例として提示される
- ユーザー側のUIに変更はない（内部の事例品質向上のみ）

## スコープ外

- LangGraphステートマシンへの移行（ver13以降）
- 検索結果の表示UIの追加（出典リンク等）
- 検索回数制限やレートリミット
