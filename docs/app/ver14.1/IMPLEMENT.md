## 既存例示の扱い

現在の `CASE_SEARCH_PROMPT` の「事例選定の基準」セクションにある例示:

```
- 元の課題とは異なる分野から選ぶこと（例: 生物の形態模倣、建築、経済、自然現象、スポーツなど）
```

この行を**削除し、新しい体系的なカテゴリセクションに置き換える**。既存の例示は新カテゴリリストに包含されるため、情報の欠落はない。

```
## 出力フォーマット
```

```
## 事例選定の基準
- 元の課題とは異なる分野から選ぶこと
- 以下のカテゴリを参考に、多様な分野から事例を選定する（全てを使う必要はない）:
  - バイオミミクリー（生物模倣）: 生物の形態・行動・生態系の仕組みからの着想（例: ヤモリの足→接着技術、ハチの巣→軽量構造）
  - 異業種ビジネスモデル: 他業界の戦略・オペレーション・収益モデル（例: 航空業界のダイナミックプライシング→ホテル業界）
  - 歴史的発明・発見: 過去の技術革新や科学的ブレイクスルーの原理（例: ワイン圧搾機の機構→グーテンベルクの印刷機への転用）
  - 芸術・デザイン: 美術・音楽・建築デザインの構成原理（例: 黄金比→UI設計）
  - 自然現象・物理法則: 物理・化学・地学のメカニズム（例: 浸透圧→組織内の情報伝達）
  - 社会システム・制度: 経済・法律・教育・医療の仕組み（例: トリアージ→タスク優先順位付け）
  - スポーツ・ゲーム戦術: 戦略・チームワーク・ルール設計（例: サッカーのゾーンディフェンス→セキュリティ対策）
- Web検索結果から得られた実在の事例を優先的に取り上げる
- 各事例について、なぜ類似しているかを一文で説明する
- 1回の応答で使うカテゴリは3〜5種類に絞り、異なるカテゴリから1つずつ事例を選ぶことで多様性を確保する

## 末尾の指示
（変更なし）

## ルール
（変更なし）
```

### 設計判断

1. **カテゴリに具体例を付与**: カテゴリ名だけでは LLM が抽象的な連想に留まる可能性があるため、各カテゴリに1〜2個の具体的なアナロジー例を括弧内に記載。これにより LLM が「このカテゴリではこのような転用がある」という具体的なイメージを持てる

2. **「全てを使う必要はない」の明記**: 7カテゴリ全てを毎回使わせると不自然な事例が混じるリスクがある。カテゴリは参照用リストであり、課題に合った3〜5種類を選択するよう指示

3. **「異なるカテゴリから1つずつ」のルール追加**: 同じカテゴリから複数事例を出すと多様性が損なわれるため、カテゴリ分散を明示的に指示

4. **7カテゴリの選定根拠**: ISSUES の候補（バイオミミクリー、異業種、歴史的発明、芸術・デザイン）に加え、元のプロンプトで既に言及されていた分野（自然現象、スポーツ）と、アナロジー思考の文献で頻出する分野（社会システム）を加えた。多すぎず少なすぎない7カテゴリ

## 品質検証

プロンプト変更は LLM 出力に直接影響するため、実装後に手動で検証を行う:

1. 開発サーバーを起動し、2〜3パターンの課題を入力
2. 提示される事例が複数の異なるカテゴリから選ばれていることを確認
3. 事例の質（関連性・具体性）が変更前と同等以上であることを確認

※ 自動テストの追加は不要（プロンプト品質は LLM の出力内容に依存するため、決定的なテストが書けない）
