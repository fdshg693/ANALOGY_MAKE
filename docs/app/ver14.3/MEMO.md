# ver14.3 MEMO

## 実装判断の乖離

### フェーズ1（診断）のスキップ

IMPLEMENT.md ではフェーズ1として診断ログを追加し、原因を確定してからフェーズ2の修正に進む計画だった。しかし、以下の理由によりフェーズ1をスキップしてフェーズ2（`isInstance` への置き換え）を直接適用した:

- `instanceof` がデシリアライズ後のオブジェクトで失敗するのは LangChain の既知のパターンであり、蓋然性が十分高い
- `isInstance` は `instanceof` の上位互換であり、正常なクラスインスタンスにも、デシリアライズ後のオブジェクトにも対応する。仮に原因が別（チェックポイント保存の問題）であっても、`isInstance` への移行自体にデメリットはない
- 診断ログの追加・確認・削除のサイクルにはサーバー起動とブラウザでの再現が必要で、非対話的な実装セッションでは効率が悪い

### 動作確認について

ローカルでの実機確認（dev サーバー起動 + ブラウザでの再現テスト）は未実施。Azure 本番環境での確認も未実施（スコープ外）。テストは全 54 ケースパス、typecheck もパス済み。

## 注意事項

### テストモックの脆弱性

新規追加テスト（`isInstance ベースの型チェックでデシリアライズ後メッセージを正しく処理`）は `HumanMessage.isInstance` / `AIMessage.isInstance` の内部実装（`Symbol.for('langchain.message')` + `type` プロパティ）に依存したモックを使用している。`@langchain/core` のバージョンアップで `isInstance` の判定ロジックが変更された場合、モック構造の見直しが必要。

### 実機確認の推奨

本修正の根本原因（`instanceof` vs デシリアライズ後オブジェクト）は実機でのみ完全に確認できる。次回のデプロイ時に以下を確認することを推奨:
1. 新しい会話を行い、ブラウザリロード後に AI メッセージが表示されるか
2. 既存のスレッドを選択して履歴が正しくロードされるか
