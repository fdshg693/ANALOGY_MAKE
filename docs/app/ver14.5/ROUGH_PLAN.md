# ver14.5 ROUGH_PLAN

`ISSUES/app/high/履歴修正-実機確認.md`

## 判断根拠

- MASTER_PLAN の全フェーズが完了済み（PHASE3 未策定）のため、ISSUES 消化フェーズ
- `ISSUES/app/high/` に未解決の高優先度 ISSUE が存在 → ISSUES 対応を優先
- ver14.4 RETROSPECTIVE でも ver14.5 での履歴バグ根本修正を推奨

## 背景

- ver14.3: `instanceof` → `isInstance` への置き換えを実施したが、履歴不具合が継続
- ver14.4: 調査基盤を整備（永続ログ機能 + SQLite 調査 CLI ツール）
- 根本原因の特定・修正が未着手

## 目的

ver14.4 で整備した調査基盤（永続ログ・SQLite CLI ツール）を活用し、履歴不具合の根本原因を特定して修正する。

## 事前リファクタリング

不要。既存コード構造は調査・修正に支障なし。

## スコープ

### やること

1. **根本原因の調査**: 永続ログと SQLite CLI ツールを使い、履歴取得時にメッセージが欠落・変質する原因を特定する
2. **根本原因の修正**: 特定した原因に基づいてコードを修正する
3. **修正の検証**: 修正後に履歴が正しく表示されることを確認する（既存テストの通過 + 必要に応じてテスト追加）

### やらないこと

- 新機能の追加
- 調査基盤そのものの拡張（ver14.4 で整備済みの範囲で対応）
- PHASE3 マスタープランの策定（履歴バグ解消後の次バージョン以降）

### スコープ調整の判断基準

調査の結果、修正がアーキテクチャ変更を伴う大規模なものになる場合は、本バージョンを「調査結果の記録 + ISSUE 内容の更新」に留め、修正は次バージョンに分割する。

## 期待される成果

- スレッド切り替え・ページリロード時に、会話履歴が正しく復元される
- `ISSUES/app/high/履歴修正-実機確認.md` の解消（または、調査結果に基づく ISSUE 内容の更新）
