### 良かった点

1. **リスク先送りの明示化パターンが確立**: IMPLEMENT.md の「リスク・不確実性」5 項目に対し、MEMO.md で「検証済み／検証不要／検証先送り」の 3 分類を全件適用。先送りにした 2 項目は `ISSUES/app/medium/` 配下に独立ファイルとして起票され、本番兆候・対応方針・影響範囲が整理されている。リスクが MEMO 内に埋もれて次バージョンで忘れられるリスクを回避した

2. **大規模計画でも計画どおりに収束**: バックエンドとフロントエンドにまたがる 6 ファイル超の変更を、IMPLEMENT.md の「実装順序」どおり段階的に進めて全テストグリーン。1〜3 の各ステップでバックエンド単体検証が可能な構造にしていたため、仮にチェックポイント永続化リスクが顕在化してもフロント着手前に切り替えられた（結果として切り替え不要だった）

3. **後方互換テストが計画段階で組み込まれた**: `history.get.ts` で ver15.x 以前のメッセージ（`additional_kwargs.searchResults` 無し）を扱う挙動について、IMPLEMENT.md 時点でテスト計画に明記 → 実装で 3 ケース（正常展開／不正形式フィルタ／旧形式）を追加。ver15.1 の `getThreadSettings` 段階マージに続く、2 回目の後方互換パターン適用

### 改善すべき点

1. **実機確認 6 バージョン連続先送り**

   ver14.3 以降、dev サーバー起動・ブラウザ動作確認なしで進行中。ver16.0 は新規コンポーネント `SearchResultsList.vue`（折りたたみ UI + ドメイン抽出ロジック）を追加しており、視覚挙動は未確認。typecheck + 93 ケースのテストで論理検証は済んでいるが、`<details>` の展開挙動・外部リンクのブラウザ挙動・ダークモード等の視覚面は非対話セッションでは確認不能。

   **影響**: 低〜中。
   **対応**: スキル変更は引き続き提案しない（ver15.0/15.1 と同方針）。ユーザーデプロイ前の手動確認を推奨。

2. **未検証リスクが ISSUES/medium に 2 件蓄積**

   `getState-timing.md` と `additional-kwargs-sqlite.md` はどちらも LangGraph の実機挙動に依存しており、ユニットテスト（モック）では原理的に検証不可。どちらも「本番で症状が出たら切り替える」フォールバックが用意されているが、検証手段として最低限「ステージング環境でスレッド切り替え → 再読み込み」を 1 回実行するだけでも解消できる類のもの。

   **影響**: 低。実害が出た場合の対応方針は整っている。
   **対応**: 次のデプロイ後、ユーザーにブラウザで「新しいスレッドで質問 → 別スレッドに切り替え → 戻って再読み込み」の動作確認を依頼すれば 2 件とも同時に解消見込み。スキル変更ではなく `REQUESTS/AI/` または MEMO 側の注記で扱う範囲。

## 3. 次バージョンの種別推奨

### 推奨: メジャーバージョン 17.0

**対象**: PHASE3 項目 3「会話分岐（メッセージ編集 → 分岐作成 → 分岐間ナビゲーション）」

**理由**:

1. **データモデル拡張**: スレッド単位ではなく「分岐ツリー」として会話を保持する必要があり、SQLite スキーマ／LangGraph state／履歴 API に広範な影響

3. **破壊的ではないが大きな新機能**: MASTER_PLAN PHASE3 の新項目着手、メジャー条件に合致

## 4. スキル改善（本ステップで即時適用）

### 4-A. imple_plan SKILL: 「検証先送り」リスクの ISSUES 化を明文化

**背景**: ver16.0 MEMO で「検証先送り → ISSUES ファイル追加」のパターンが自発的に適用され、リスクが MEMO 内に埋もれる問題を解消した。ver15.1 までは「MEMO に記載する」で終わっていたため、次バージョンでリスクが見落とされる可能性があった。このパターンをスキル本体に恒久反映する。

**変更**: `.claude/skills/imple_plan/SKILL.md` の「動作確認」セクション（リスク・不確実性の処理箇所）に、検証先送り時は `ISSUES/{カテゴリ}/{優先度}/` に独立ファイルを作成することを明記する。

**即時適用**: 本 retrospective 内で編集済み。

### その他の提案なし

ver16.0 のフローは split_plan〜write_current まで既存スキルで円滑に流れており、追加の改善提案はない。
