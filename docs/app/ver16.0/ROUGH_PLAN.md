# ver16.0 ROUGH_PLAN

## 対象

MASTER_PLAN PHASE3 項目2「検索結果の可視化（Tavily 検索の生データ表示）」

## 背景

現状 `caseSearchNode` は Tavily Web 検索を実行し、結果文字列を LLM のシステムプロンプトに埋め込むだけで、検索結果そのものはユーザーには届かない。ユーザーには「AI がどのような外部情報を参考にして事例を提示したか」が見えず、以下の不都合がある:

- 提示された事例の根拠・出典が不明で、信頼性を判断できない
- Web 検索 ON／OFF（ver15.1 実装）を切り替えた際の効果が実感しにくい
- 抽象化された課題に対する Tavily の検索精度を評価できない（開発者視点）

## 機能スコープ

**caseSearch ノード実行時に取得した Tavily 検索結果を、AI 応答と併せてフロントエンドに届け、UI 上で閲覧可能にする。**

### 含むもの

1. **検索結果データのフロントエンド到達**
   - caseSearch ノードの Tavily 呼び出しで得られた結果（タイトル・URL・スニペット等）を、ストリーミング経路を通じてクライアントに配信する
   - 会話履歴の一部として保存し、後から同じスレッドを開いても再表示できる

2. **UI での検索結果表示**
   - アシスタントメッセージ（事例提示）に付随するかたちで、「参考にした Web 検索結果」を折りたたみ UI で表示する
   - 各検索結果は、タイトル（外部リンク化）・ドメイン・スニペット（短い抜粋）を並べる
   - Web 検索 OFF の場合、または検索結果が空の場合は表示しない
   - 検索実行中は「検索中…」の軽いフィードバックを出す（必須ではなく、実装コストが低ければ含める）

3. **スレッド切り替え・履歴復元時の整合性**
   - 過去の会話を開き直したとき、その会話時点で取得された検索結果が同じメッセージに紐づいて復元される

### 含まないもの（将来バージョン送り）

- 検索結果からのフォローアップ質問や、特定結果の「お気に入り」機能
- 検索結果のフィルタリング・ソート UI
- 検索結果をベースにした分岐作成（PHASE3 項目3「会話分岐」で扱う）
- Tavily 以外の検索ソース統合
- 古いスレッド（ver15.x で作成された、検索結果を持たないスレッド）への遡及的な補完

## ユーザー体験の変化

**Before (ver15.1):**
- ユーザーが課題を送信 → AI が事例を提示（Web 検索結果は裏でプロンプトに注入されているが UI には表れない）

**After (ver16.0):**
- ユーザーが課題を送信 → AI が事例を提示 → 事例メッセージの下に「参考にした Web 検索結果 (N 件)」の折りたたみセクション → クリックで展開、各結果のタイトル・URL・スニペットが見える
- スレッドを開き直しても同じ表示が復元される
- Web 検索 OFF のスレッドでは当該セクション自体が出ない

## 前提条件

**必須（ブロッカー）**:
- ver15.1 の検索設定機能がそのまま動作していること（`search.enabled` が `false` のときに検索がスキップされる挙動は維持される）
- Tavily API から `title` / `url` / `content`（snippet）の 3 フィールド相当が取得できること（@langchain/tavily の戻り値を IMPLEMENT 段階で型確認）

**推奨（進行可能だが確認が望ましい）**:
- ver15.0〜15.1 で 5 バージョン連続先送りされている dev サーバー実機確認。ver16.0 は UI 追加（折りたたみ表示）を伴うため、少なくとも実装完了後に 1 度は手元で開いて挙動を確認するのが望ましい
- Tavily の無料プラン呼び出し制限を考慮した検索呼び出し頻度（既存の 1 スレッド 1 検索と変わらない見込みだが、過去スレッド遡及の有無次第で変わる）

## メジャーバージョン判定

以下の条件に該当するため **ver16.0（メジャー）** とする:
- MASTER_PLAN PHASE3 の新項目（項目2）に着手する
- SSE ストリーミングのイベント種別を新設する（`token` / `done` / `error` に加えて検索結果配信イベント）必要があり、データフローが拡張される
- LangGraph ステートに検索結果を保持する新しいフィールドを追加する可能性があり、履歴復元経路にも影響が及ぶ
- 新規フロントエンドコンポーネントを作成する

## 小規模タスク判定

**該当しない**。変更対象ファイル数は、バックエンド（agent, chat.post, history.get, thread-store または state 定義）とフロントエンド（新規 SearchResults コンポーネント、ChatMessage または useChat の拡張、SSE パーサーの拡張）で合計 6 ファイル以上の見込み。追加行数も 200 行を超える見込み。新規ファイル作成あり。→ フルワークフローで継続する。

## 事前リファクタリング判断

**事前リファクタリング不要**。現在の構造（LangGraph ステート + `messagesStateReducer` による履歴永続化、h3 SSE による 1 方向ストリーミング、`AIMessage` への `additional_kwargs` 添付、`parseSSEStream` の event 拡張可能な構造）は、検索結果という補助データの追加を素直に受け入れられる。既存の `performSearch()` の戻り値型変更は実装の一部として扱う。
