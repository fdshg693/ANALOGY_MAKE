## タスク概要

ユーザーが過去の自分のメッセージを編集することで、会話を**分岐**させられるようにする。分岐元のチェックポイントから新しい会話履歴が再生成され、元の会話履歴も保持される。ユーザーはメッセージ上に表示される分岐ナビゲーションで、異なる分岐を自由に行き来できる。

### ユーザー体験の変化

**変更前（ver16.1 時点）**:

- 過去のメッセージは読むだけ。一度送信した発言は修正できない
- 「別の方向で話を進めたい」と思っても、新規スレッドを作るか、追加メッセージで軌道修正するしかない

**変更後（ver17.0）**:

- 過去の自分のメッセージにホバーすると「編集」ボタンが表示される
- 編集して送信すると、その位置から**新しい分岐**として会話が再生成される。元の会話も残る
- 編集したメッセージの横に `◀ 1/2 ▶` のような分岐ナビゲーションが表示され、分岐間を自由に切り替えられる
- 分岐を切り替えると、それ以降のメッセージ表示もその分岐の内容に切り替わる
- スレッド一覧での「現在どの分岐を見ているか」の状態はスレッド全体の設定として保持される（リロード後も復元）

### スコープに含めるもの

- ユーザーメッセージの編集 UI（ホバーで編集ボタン表示 → テキストエリア編集 → 送信）
- 分岐データモデル（新規テーブル `thread_branches`）と永続化
- 分岐作成 API（`POST /api/chat/fork`）と分岐一覧取得 API（`GET /api/chat/branches`）
- 既存の履歴取得 API（`GET /api/chat/history`）に `branchId` パラメータを追加
- 分岐ナビゲーション UI（◀ N/M ▶）
- 分岐切替時の履歴ロード・SSE 送信時の分岐紐付け
- アクティブな分岐 ID のスレッド設定への保存（リロード後の復元のため）

### スコープ外

- **AI メッセージの編集**: 今回はユーザーメッセージのみ対象（AI 回答の書き換えは別機能）
- **分岐の削除・統合**: ユーザーによる分岐削除 UI は提供しない（DB には残り続ける）
- **分岐の命名・ラベリング**: 分岐は連番で識別（`1/N` 表示）。タイトル付け UI は提供しない
- **分岐元より前のメッセージ編集時の整合性**: 分岐は「編集したメッセージの次から」再生成するのみ。過去に遡って複数箇所を編集する操作は UI 上できない
- **分岐間のメッセージコピー・マージ**: 提供しない
- **エコーモード中の分岐**: エコーモードでも分岐作成は可能にするが、専用 UX は作り込まない（既存のエコー SSE 経路をそのまま再利用）

## 前提条件・依存関係

- LangGraph の `SqliteSaver` は各ステップのチェックポイントを既に保存している（ver14.0 以降）
- チェックポイントの参照・再実行（`Command({ resume: ... })` 相当）の LangGraph API 挙動は、実装計画段階で `@langchain/langgraph` の最新ドキュメントで要確認
- 既存テーブル `threads`（`settings` カラムを含む）と LangGraph のチェックポイントテーブル（`checkpoints`, `writes`, `checkpoint_blobs` 等）は変更しない
- 新規テーブル `thread_branches` を追加するのみ（マイグレーション処理: アプリ起動時に `CREATE TABLE IF NOT EXISTS`）

## 成功基準

- ユーザーがメッセージを編集 → 新しい分岐が生成 → `◀ 1/2 ▶` で元の分岐に戻れる
- ページリロード後もアクティブな分岐が復元される
- 既存スレッド（ver16.x 以前で作成されたもの）は `branchId = 'main'` で暗黙扱いされ、従来通り動作する
- ユニットテスト: 分岐作成 / 分岐切替 / 履歴取得（`branchId` 指定）の新規ケースをカバー
- `pnpm test` / `npx nuxi typecheck` がグリーン

## 事前リファクタリング要否

「会話分岐」は既存の `chat.post.ts` / `history.get.ts` / `ChatMessage.vue` 等に新しい分岐概念を貫通させる必要がある。現状の `threadId` 一本足の設計を `(threadId, branchId)` の二本立てに拡張する影響が大きいため、**事前リファクタリング（`REFACTOR.md`）を作成して整理する**。

具体的には以下の論点を REFACTOR で扱う:

- 既存コードの `threadId` のみを鍵にしている箇所の洗い出し（履歴取得・SSE 送信・フロント状態）
- `branchId` 必須化 vs デフォルト `'main'` の扱い方針
- `useChat.ts` / `useSettings.ts` とは別に `useBranches.ts` を切る設計の整合性確認
- ver16.1 RETROSPECTIVE で指摘された「`ThreadSettings` のテストフィクスチャ更新コスト」についてのテストヘルパ導入は、本バージョンで取り込むかを判断（`ThreadSettings` にさらにフィールドが増える見込みのため、このタイミングで `{ ...DEFAULT_SETTINGS, ...override }` パターンに整理することを検討）
- 「編集ボタンをどこまで遡って表示するか」の方針決定（すべてのユーザーメッセージに付与するか、最古の分岐元とする下限を設けるか）
