# ver3.3 IMPLEMENT

## 設計方針

CURRENT.md をインデックス + 詳細ファイルに分割する構成を導入する。既存の単一ファイル形式との後方互換性を維持する。

### ファイル構成

```
docs/{カテゴリ}/ver{X.0}/
  CURRENT.md              ← インデックス（全体構成・各セクションの概要 + リンク）
  CURRENT_skills.md       ← 詳細: SKILLファイルの説明
  CURRENT_scripts.md      ← 詳細: スクリプトの説明
  ...                     ← トピックごとに分割
```

### 分割基準

- CURRENT.md が **150行を超える場合**、トピック単位で詳細ファイルに分割する
- 各詳細ファイルは **50〜200行** を目安とする
- 150行以下なら従来通り単一ファイルで記述してよい

### 命名規則

- インデックス: `CURRENT.md`（変更なし）
- 詳細ファイル: `CURRENT_{トピック名}.md`（例: `CURRENT_skills.md`、`CURRENT_scripts.md`）

## 変更内容

### 1. write_current/SKILL.md

**現状** (L39-48): CURRENT.md の記載ルールに分割の記述が1行のみ（L48）

**変更**: 「CURRENT.md の記載ルール」セクションを以下のように書き換える。旧 L48 の「`docs/{カテゴリ}` 配下に機能単位のドキュメントを作成して分割」という曖昧な記述は削除し、新設の「ファイル分割」セクション（バージョンフォルダ内に `CURRENT_{トピック名}.md` を作成）で完全に置き換える:

```markdown
#### CURRENT.md の記載ルール

- **ASIS で書く**（変更点・差分ではなく、現時点のコード状況を記述する）
- **CLAUDE.md と重複する内容は記載しない**
- CURRENT.md には **CLAUDE.md がカバーしない粒度の詳細**を ASIS で記述する:
  - 各ファイルの現在の役割・実装内容（ファイル名・行数・概要）
  - API契約の詳細（リクエスト・レスポンス形式など）
  - 技術的な決定事項の詳細（採用理由・代替案との比較など）
  - 未実装の課題

#### CURRENT.md のファイル分割

- **分割基準**: CURRENT.md が **150 行を超える場合**、トピック単位で分割する。150 行以下なら単一ファイルのまま記述してよい
- **命名規則**: `CURRENT_{トピック名}.md`（例: `CURRENT_skills.md`、`CURRENT_scripts.md`）
- **インデックス構成**: 分割時の `CURRENT.md` は以下の構成とする:
  1. 冒頭にバージョンタイトルとカテゴリの全体概要（1〜3 行）
  2. 各詳細ファイルへのリンクと、そのファイルが扱う範囲の1行説明
  3. 分割するほどでない小さなセクション（設定ファイル・`.gitignore` 等）はインデックス内に直接記載してよい
- **各詳細ファイル**: 50〜200 行を目安とする
```
