### 良かった点

- **スコープ切り出しの判断が明瞭**: ROUGH_PLAN で「PHASE4.0 全体ではなく model/effort 指定のみ。セッション継続は ver4.1 へ」と明記し、理由（CLI フラグ未検証・不確実性）まで添えた。後工程が迷わず、想定外の膨張を防げた

### 改善が必要な点

#### 2-1. IMPLEMENT.md の「リスク・不確実性」が動作確認で放置される

ver4.0 のリスク 7-1（`--effort` フラグの CLI 受理確認）は IMPLEMENT.md で明確に flag されていたにもかかわらず、`imple_plan` の動作確認では **未実施のまま MEMO.md に「未実施」と書かれて終了** した。MEMO の記述は:

> 実走での CLI 受理確認は未実施。… 受理されない場合は `build_command` の `--effort` 付与部分を一時コメントアウトすれば回避可能

これは「問題が起きたら後で考える」であり、リスクを受け入れたのか先送りしたのかが曖昧。`imple_plan` SKILL の動作確認セクションに「`リスク・不確実性` セクションの各項目は検証／不要／先送りのいずれかを明記」を追加することで、次回以降は同様の漏れを防ぐ。

**適用済み**: `.claude/SKILLS/imple_plan/SKILL.md` 動作確認セクション 4 項目目に追記（本ステップで即時適用）。

#### 2-2. wrap_up ステップで「コミット無し」のケースが不可視

MEMO に実対応事項がなく wrap_up がコード/ドキュメント変更を伴わなかった場合、SKILL の手順 5（コミット）で「コミット対象がないので何もしない」という扱いが明示されていなかった。結果として:

- wrap_up が実行されたのか、省略されたのか、ログ/コミット履歴から判別不能
- plan_review_agent への報告手順（「対応不要」「先送り」時）が空実行の場合にも必要なのか曖昧

**適用済み**: `.claude/SKILLS/wrap_up/SKILL.md` 手順 5 に「コミット対象の変更が存在しない場合はスキップ可・MEMO 末尾または進捗報告に『wrap_up: 変更なし（スキップ）』と明記」を追記。

#### 2-3. MEMO.md の「未対応」と「リスク」の切り分け

ver4.0 の MEMO には:

- 「未対応（ver4.1 以降）」セクション → ROUGH_PLAN で宣言済みの非対象スコープ
- 「リスク 7-1」セクション → 未検証リスク（非対象ではなく、確認漏れ）

が混在していて、wrap_up で後者を検出しにくくなっていた。今回の SKILL 改善（2-1）で「リスク・不確実性」項目を動作確認に組み込むことで、MEMO 内でも分離が進むはず。

### SKILL への即時適用（本ステップで実施）

| `.claude/SKILLS/imple_plan/SKILL.md` | 動作確認セクションに「IMPLEMENT.md の `## リスク・不確実性` の各項目は検証／不要／先送りのいずれかを MEMO.md に記録する」を追加 |

## 3. 次バージョンの種別推奨

### 次バージョン候補の棚卸し

| 候補 | 内容 | スコープ規模 | 種別判定 |
|---|---|---|---|
| A. `ISSUES/util/medium/スクリプト改善.md` 対応 | `scripts/README.md` 新規作成 + `claude_loop.py` のファイル分割 | 変更ファイル 3〜5、主に新規追加・分割 | **マイナー (4.1)** — 既存機能の整理、アーキテクチャ不変 |
| B. PHASE4.0 残タスク（セッション継続） | `-r` / `--session-id` / stream-json パーサ追加、`_run_steps` に session state 導入 | ステップ間 state と stdout パーサが新規、CLI フラグ複数の互換確認が必要 | **メジャー (5.0)** — アーキテクチャ変更（ステップ間で状態を引き継ぐ新機構）に該当 |
| C. PHASE5.0（ISSUE ステータス管理） | frontmatter ベースのステータス遷移、`/issue_review` 共通 SKILL 新規作成 | SKILL 新規追加 + 既存 SKILL 2 件の冒頭フェーズ追加 | **メジャー (5.0)** — MASTER_PLAN 新項目着手 |

### 推奨

**第一推奨: A（ver4.1 マイナー）**

理由:
- `scripts/claude_loop.py` が 698 行に成長しており、PHASE4.0 残タスク（B）や PHASE5.0（C）でさらに拡張するとメンテ負荷が高まる。**次の大きな変更の前に分割しておくのが得策**
- ROUGH_PLAN.md は小規模タスクとして quick ワークフローで消化可能（変更ファイル 3 つ程度・新規 README 以外は既存コード移動が主）
- 付随して「YAML フォーマット説明」の README を用意すれば、PHASE4.0 残タスク時に YAML スキーマ拡張のドキュメントも追加しやすい

**第二推奨: B（ver5.0 メジャー）**

理由:
- A を飛ばして直接 PHASE4.0 を閉じにいくケース。ただし現在の `claude_loop.py` 単一ファイル構成での増改築はコスト高
- 着手前に「A を先に終わらせる」判断を改めて行うのが安全

**非推奨: C（ver5.0 メジャーだがタイミングが悪い）**

理由:
- PHASE5.0（ISSUE ステータス管理）は独立に進められるが、`/split_plan` / `/quick_plan` への組み込みが必要で SKILL 側の変更量が大きい
- PHASE4.0 の残タスク（B）を先に閉じた方が、MASTER_PLAN のステータス管理が綺麗になる

### 結論

**次は ver4.1（マイナー）で `ISSUES/util/medium/スクリプト改善.md` 対応を推奨**。quick ワークフローで完結させ、その次に ver5.0 として PHASE4.0 残タスクまたは PHASE5.0 を進める。
